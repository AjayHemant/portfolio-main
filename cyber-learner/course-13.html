OWASP Top 10

Master the most critical web application security risks

Introduction to OWASP Top 10

The OWASP Top 10 is a standard awareness document for developers and web application security. It represents a broad consensus about the most critical security risks to web applications.

What is OWASP?

  • Open Web Application Security Project: Nonprofit foundation
  • Mission: Improve software security worldwide
  • Community-driven: Volunteers and security professionals
  • Free Resources: Tools, documentation, and standards
  • Global Impact: Widely adopted security framework

OWASP Top 10 2021 Overview

Top 3 Risks

  • A01 - Broken Access Control
  • A02 - Cryptographic Failures
  • A03 - Injection

Middle Risks

  • A04 - Insecure Design
  • A05 - Security Misconfiguration
  • A06 - Vulnerable Components

Lower Risks

  • A07 - Identity & Auth Failures
  • A08 - Software & Data Integrity
  • A09 - Security Logging Failures
  • A10 - Server-Side Request Forgery

A01 - Broken Access Control

Access control enforces policy such that users cannot act outside of their intended permissions. Failures typically lead to unauthorized information disclosure, modification, or destruction of data.

# Common Access Control Vulnerabilities:

1. Violation of Principle of Least Privilege:
   - Users granted more access than needed
   - Default accounts with excessive permissions
   - Administrative functions exposed to regular users

2. Bypassing Access Control Checks:
   - Modifying URL parameters
   - Changing POST to GET requests
   - Manipulating metadata or tokens

3. Elevation of Privilege:
   - Acting as admin without being logged in
   - Accessing functions as regular user that require admin

4. Metadata Manipulation:
   - Tampering with JWT tokens
   - Modifying cookies or hidden fields
   - Replaying access control tokens

# Examples:
URL Manipulation:
https://example.com/user/profile?id=123
https://example.com/user/profile?id=456  # Access other user's profile

Parameter Tampering:
POST /transfer HTTP/1.1
amount=100&from=123&to=456
# Change 'from' parameter to transfer from another account

Force Browsing:
https://example.com/admin/panel  # Direct access to admin functions

Prevention Strategies

  • Deny by Default: Implement access control with default deny
  • Centralized Access Control: Use consistent mechanisms throughout application
  • Server-side Validation: Never rely on client-side access controls
  • Logging and Monitoring: Log access control failures and alert administrators
  • Rate Limiting: Implement rate limiting for API and login attempts

A02 - Cryptographic Failures

Previously known as "Sensitive Data Exposure," this category focuses on failures related to cryptography which often leads to sensitive data exposure or system compromise.

# Common Cryptographic Failures:

1. Transmitting Data in Clear Text:
   - HTTP instead of HTTPS
   - Unencrypted email protocols
   - Database connections without encryption
   - File transfers without encryption

2. Using Old or Weak Cryptographic Algorithms:
   - MD5, SHA1 for passwords
   - DES, 3DES encryption
   - RC4 stream cipher
   - Weak random number generators

3. Default or Weak Cryptographic Keys:
   - Default encryption keys
   - Weak key generation
   - Improper key management
   - Hard-coded cryptographic keys

4. Missing Encryption:
   - Passwords stored in plain text
   - Sensitive data not encrypted at rest
   - Backup files not encrypted
   - Log files containing sensitive data

# Examples:
Weak Password Hashing:
// Bad
password = md5(user_password)

// Good
password = bcrypt(user_password, salt, rounds=12)

Insecure Data Transmission:
// Bad
http://example.com/login

// Good
https://example.com/login

Weak Encryption:
// Bad
cipher = DES.new(key, DES.MODE_ECB)

// Good
cipher = AES.new(key, AES.MODE_GCM)

Protection Measures

  • Classify Data: Identify sensitive data and apply appropriate controls
  • Encrypt in Transit: Use TLS for all sensitive data transmission
  • Encrypt at Rest: Encrypt sensitive data stored in databases and files
  • Strong Algorithms: Use up-to-date and strong encryption algorithms
  • Proper Key Management: Implement secure key generation, storage, and rotation

A03 - Injection

Injection flaws occur when untrusted data is sent to an interpreter as part of a command or query. The attacker's hostile data can trick the interpreter into executing unintended commands.

# Types of Injection Attacks:

1. SQL Injection:
   - Classic SQL injection
   - Blind SQL injection
   - Time-based SQL injection
   - Union-based SQL injection

2. NoSQL Injection:
   - MongoDB injection
   - CouchDB injection
   - Redis injection

3. OS Command Injection:
   - Shell command injection
   - Code injection
   - File inclusion attacks

4. LDAP Injection:
   - Authentication bypass
   - Information disclosure

# SQL Injection Examples:
Vulnerable Code:
query = "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'"

Attack Payload:
username: admin'--
password: anything

Resulting Query:
SELECT * FROM users WHERE username='admin'--' AND password='anything'

# Command Injection Example:
Vulnerable Code:
system("ping -c 4 " + user_input)

Attack Payload:
user_input: 8.8.8.8; cat /etc/passwd

Resulting Command:
ping -c 4 8.8.8.8; cat /etc/passwd

Prevention Techniques

Input Validation

  • Whitelist validation
  • Input sanitization
  • Data type validation
  • Length restrictions

Parameterized Queries

  • Prepared statements
  • Stored procedures
  • ORM frameworks
  • Query builders

Output Encoding

  • HTML encoding
  • URL encoding
  • JavaScript encoding
  • CSS encoding

A04 - Insecure Design

Insecure design is a broad category representing different weaknesses, expressed as "missing or ineffective control design." It focuses on risks related to design and architectural flaws.

# Insecure Design Examples:

1. Missing Security Controls:
   - No rate limiting on password reset
   - Missing multi-factor authentication
   - No account lockout mechanisms
   - Insufficient logging and monitoring

2. Weak Security Controls:
   - Security questions with predictable answers
   - Weak password policies
   - Insufficient session timeout
   - Poor error handling revealing system information

3. Business Logic Flaws:
   - Price manipulation in e-commerce
   - Workflow bypass vulnerabilities
   - Race condition vulnerabilities
   - Time-of-check to time-of-use (TOCTOU) issues

# Example - Insecure Password Reset:
1. User requests password reset
2. System sends reset link via email
3. Reset link has no expiration time
4. Reset link can be used multiple times
5. No rate limiting on reset requests

# Secure Design Alternative:
1. User requests password reset
2. System generates time-limited token (15 minutes)
3. Token can only be used once
4. Rate limiting prevents abuse
5. Account lockout after multiple failed attempts
6. Logging and monitoring of reset attempts

Secure Design Principles

  • Defense in Depth: Multiple layers of security controls
  • Fail Securely: System fails to a secure state
  • Principle of Least Privilege: Minimum necessary access
  • Separation of Duties: Critical operations require multiple approvals
  • Zero Trust Architecture: Never trust, always verify

A05 - Security Misconfiguration

Security misconfiguration is commonly a result of insecure default configurations, incomplete configurations, open cloud storage, misconfigured HTTP headers, and verbose error messages.

# Common Misconfigurations:

1. Default Configurations:
   - Default passwords not changed
   - Default accounts enabled
   - Sample applications not removed
   - Default error pages revealing information

2. Unnecessary Features Enabled:
   - Unused services running
   - Unnecessary ports open
   - Debug mode enabled in production
   - Directory listing enabled

3. Missing Security Headers:
   - Missing HSTS headers
   - No Content Security Policy
   - Missing X-Frame-Options
   - No X-Content-Type-Options

4. Improper Error Handling:
   - Stack traces in error messages
   - Database errors exposed to users
   - System information in error pages
   - Verbose logging in production

# Security Headers Examples:
HTTP/1.1 200 OK
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src 'self'
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
X-XSS-Protection: 1; mode=block
Referrer-Policy: strict-origin-when-cross-origin

# Secure Configuration Checklist:
- Remove default accounts and passwords
- Disable unnecessary services and features
- Configure proper error handling
- Implement security headers
- Regular security updates and patches
- Secure file permissions
- Network segmentation
- Regular configuration reviews

Configuration Management

  • Hardening Guidelines: Follow security hardening standards
  • Configuration Management: Use automated configuration tools
  • Regular Reviews: Periodic security configuration audits
  • Minimal Installation: Install only necessary components
  • Segmentation: Separate environments and network segments

A06 - Vulnerable and Outdated Components

Components run with the same privileges as the application itself, so flaws in any component can result in serious impact. Such flaws may be accidental or intentional.

# Component Vulnerability Issues:

1. Unknown Component Inventory:
   - No inventory of client and server-side components
   - Unknown versions of components
   - Nested dependencies not tracked
   - Third-party libraries not monitored

2. Outdated or Vulnerable Components:
   - Operating system vulnerabilities
   -Framework vulnerabilities (Struts, Spring, etc.)
   - Library vulnerabilities (jQuery, Bootstrap, etc.)
   - Database vulnerabilities
   - Web server vulnerabilities

3. Unsupported Components:
   - End-of-life software
   - No security patches available
   - Vendor no longer supports product
   - Community-maintained projects abandoned

# Example Vulnerable Components:
Apache Struts 2 (CVE-2017-5638):
- Remote code execution vulnerability
- Affected many major organizations
- Exploited in Equifax breach

jQuery < 3.0.0:
- Cross-site scripting vulnerabilities
- DOM manipulation issues
- Prototype pollution attacks

# Component Management Tools:
Dependency Scanners:
- OWASP Dependency-Check
- Snyk
- WhiteSource
- Black Duck
- Sonatype Nexus

Package Managers:
- npm audit (Node.js)
- pip-audit (Python)
- bundle audit (Ruby)
- composer audit (PHP)

Component Security Practices

  • Inventory Management: Maintain complete component inventory
  • Vulnerability Monitoring: Subscribe to security advisories
  • Regular Updates: Keep components up to date
  • Risk Assessment: Evaluate component security before adoption
  • Automated Scanning: Use tools to identify vulnerable components

A07 - Identification and Authentication Failures

Previously known as "Broken Authentication," this category includes weaknesses related to user identity confirmation, authentication, and session management.

# Authentication Vulnerabilities:

1. Weak Password Policies:
   - No complexity requirements
   - Common passwords allowed
   - No password history
   - Weak password recovery

2. Credential Stuffing:
   - No protection against automated attacks
   - No account lockout mechanisms
   - No CAPTCHA implementation
   - Weak rate limiting

3. Session Management Issues:
   - Predictable session IDs
   - Session fixation vulnerabilities
   - Sessions not invalidated on logout
   - Long session timeouts

4. Multi-Factor Authentication Bypass:
   - MFA not enforced for all users
   - Weak MFA implementation
   - MFA bypass vulnerabilities
   - Recovery codes not properly protected

# Examples:
Session Fixation:
1. Attacker gets session ID: JSESSIONID=123456
2. Attacker tricks user to login with this session ID
3. User authenticates successfully
4. Attacker uses the same session ID to access user's account

Credential Stuffing Attack:
1. Attacker obtains username/password lists from breaches
2. Automated tools try these credentials on target site
3. Successful logins grant access to user accounts
4. No rate limiting allows rapid testing

# Secure Authentication Implementation:
Password Policy:
- Minimum 8 characters
- Mix of uppercase, lowercase, numbers, symbols
- Check against common password lists
- Implement password history

Session Management:
- Generate cryptographically strong session IDs
- Regenerate session ID after login
- Implement proper session timeout
- Secure session storage

Authentication Best Practices

Password Security

  • Strong password policies
  • Password complexity requirements
  • Common password blacklists
  • Secure password storage (bcrypt)

Multi-Factor Authentication

  • TOTP (Time-based OTP)
  • SMS verification (with limitations)
  • Hardware tokens
  • Biometric authentication

Session Management

  • Secure session ID generation
  • Session regeneration
  • Proper session timeout
  • Secure cookie attributes

A08 - Software and Data Integrity Failures

This new category focuses on making assumptions related to software updates, critical data, and CI/CD pipelines without verifying integrity.

# Integrity Failure Examples:

1. Insecure CI/CD Pipelines:
   - No code signing verification
   - Insecure artifact repositories
   - Compromised build environments
   - No integrity checks on dependencies

2. Auto-Update Mechanisms:
   - Updates without signature verification
   - Insecure update channels (HTTP)
   - No rollback mechanisms
   - Compromised update servers

3. Insecure Deserialization:
   - Untrusted data deserialization
   - No integrity verification
   - Remote code execution via serialized objects
   - Tampering with serialized data

4. CDN and Third-Party Content:
   - No Subresource Integrity (SRI)
   - Compromised CDN content
   - Malicious third-party scripts
   - No content verification

# Subresource Integrity Example:
<script src="https://cdn.example.com/library.js"
        integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8wC"
        crossorigin="anonymous"></script>

# Secure Deserialization:
// Insecure
Object obj = deserialize(untrusted_data);

// Secure
if (verify_signature(untrusted_data)) {
    Object obj = deserialize(untrusted_data);
} else {
    throw new SecurityException("Invalid signature");
}

Integrity Protection Measures

  • Digital Signatures: Sign software updates and critical data
  • Subresource Integrity: Verify third-party content integrity
  • Secure CI/CD: Implement secure build and deployment pipelines
  • Dependency Verification: Verify integrity of dependencies
  • Secure Deserialization: Validate serialized data before processing

A09 - Security Logging and Monitoring Failures

Insufficient logging and monitoring, coupled with missing or ineffective integration with incident response, allows attackers to further attack systems and maintain persistence.

# Logging and Monitoring Failures:

1. Insufficient Logging:
   - Login attempts not logged
   - Failed access attempts ignored
   - Administrative actions not recorded
   - Security events not captured

2. Inadequate Log Protection:
   - Logs stored in plain text
   - No log integrity protection
   - Logs accessible to unauthorized users
   - No log retention policies

3. Missing Monitoring:
   - No real-time alerting
   - Security events not monitored
   - No anomaly detection
   - Delayed incident response

4. Poor Log Quality:
   - Missing context information
   - No correlation between events
   - Insufficient detail for investigation
   - No standardized log format

# Essential Security Events to Log:
Authentication Events:
- Login attempts (success/failure)
- Password changes
- Account lockouts
- Privilege escalations

Authorization Events:
- Access to sensitive resources
- Permission changes
- Administrative actions
- Failed authorization attempts

Input Validation Events:
- Injection attempts
- Malformed requests
- Suspicious input patterns
- Rate limiting triggers

# Log Format Example:
{
  "timestamp": "2023-10-15T10:30:00Z",
  "event_type": "authentication_failure",
  "user_id": "user123",
  "source_ip": "192.168.1.100",
  "user_agent": "Mozilla/5.0...",
  "details": "Invalid password attempt",
  "risk_score": 7
}

Effective Logging Strategy

  • Comprehensive Coverage: Log all security-relevant events
  • Centralized Logging: Aggregate logs from all systems
  • Real-time Monitoring: Implement automated alerting
  • Log Protection: Secure log storage and transmission
  • Incident Response: Integrate with incident response procedures

A10 - Server-Side Request Forgery (SSRF)

SSRF flaws occur whenever a web application is fetching a remote resource without validating the user-supplied URL. It allows an attacker to coerce the application to send a crafted request to an unexpected destination.

# SSRF Attack Scenarios:

1. Internal Network Scanning:
   - Scan internal IP ranges
   - Discover internal services
   - Access internal APIs
   - Bypass network firewalls

2. Cloud Metadata Access:
   - AWS metadata service (169.254.169.254)
   - Azure metadata service
   - Google Cloud metadata
   - Extract sensitive credentials

3. File System Access:
   - Read local files (file://)
   - Access configuration files
   - Read application source code
   - Extract sensitive data

4. Protocol Exploitation:
   - HTTP/HTTPS requests
   - FTP protocol abuse
   - LDAP queries
   - Database connections

# SSRF Examples:
URL Parameter Manipulation:
https://example.com/fetch?url=http://internal-server/admin

Cloud Metadata Attack:
https://example.com/fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/

File System Access:
https://example.com/fetch?url=file:///etc/passwd

# Vulnerable Code Example:
def fetch_url(url):
    response = requests.get(url)  # No validation
    return response.content

# Secure Implementation:
def fetch_url(url):
    # Validate URL scheme
    if not url.startswith(('http://', 'https://')):
        raise ValueError("Invalid URL scheme")
    
    # Parse and validate URL
    parsed = urlparse(url)
    
    # Block private IP ranges
    if is_private_ip(parsed.hostname):
        raise ValueError("Private IP not allowed")
    
    # Block metadata services
    if parsed.hostname == '169.254.169.254':
        raise ValueError("Metadata service blocked")
    
    response = requests.get(url, timeout=5)
    return response.content

SSRF Prevention Techniques

Input Validation

  • URL scheme validation
  • Domain whitelist
  • IP address filtering
  • Port restrictions

Network Controls

  • Network segmentation
  • Firewall rules
  • Proxy servers
  • DNS filtering

Application Controls

  • Response validation
  • Timeout settings
  • Error handling
  • Logging and monitoring
